# Proposal: Video Calls (LiveKit self-hosted)

## Change Name

`video-calls`

## Intent

The booking flow ends with both parties connected by status — `PENDIENTE → CONFIRMADA → EN_CURSO → COMPLETADA` — but today there is no actual channel for the doctor and the patient to meet. `DoctorHero` even carries a `tel:` "Llamar" button, which is the platform admitting, out loud, that the real consultation is happening off-platform. That is a product hole, not a feature.

This change adds a real, in-platform video consultation to the appointment flow. The change is **greenfield at the spec and code level** (no existing `video-calls` spec, no LiveKit wiring yet) and it lands **on top of the existing status machine**, not as a replacement for it. The doctor still clicks "Iniciar consulta" to move the cita to `EN_CURSO`; the call is a feature that becomes available at that point. When the call ends, the doctor still clicks "Completar" or "No asistio" to finalize. The video channel is plumbing; the status machine is the source of truth for billing and audit.

We use **LiveKit self-hosted via Docker** to keep consultation traffic in-house. The team has explicitly chosen self-hosting over LiveKit Cloud, Daily.co, or Whereby to avoid the per-minute cloud billing of competitor platforms and to keep the patient's audio/video on infrastructure the platform operates. `@livekit/components-react` and `livekit-client` are already in `package.json` from earlier prep work; the server-side SDK (`livekit-server-sdk`) is the only new dependency.

The previous change (`2026-06-16-home-page-upgrade`) was UI-only on a public page. This change is the platform's first interactive multi-party feature and the first one to introduce a non-Postgres, non-Redis, non-S3 runtime dependency (the LiveKit SFU).

## Default Decisions

These 10 decisions were committed during explore. The spec and design phases MUST honor them verbatim.

### D1 — Server-side derived room name, `cita-${uuid}`

**Decision:** The LiveKit room name is `cita-${cita.id}`, derived on the server inside `Cita.livekitRoomName`. The DB column `citas.livekit_room_name` is added for future flexibility (explicit override, audit, persistence) but is NOT the source of truth for the live room name used in token grants.

**Rationale:** Hides the naming convention from clients (prevents enumeration attacks like "iterate `cita-1` to `cita-10000`"). Keeps the client simple — it never has to construct or even know the room name; the server hands it back as part of the token grant. The DB column is reserved for a future where we may want to support ad-hoc room names (group sessions, breakout rooms) without breaking the API.

**Alternative considered:** Store the room name explicitly on every `Cita` row on creation — rejected. The derivation is trivial and adding a write-on-create column means a second source of truth that can drift.

### D2 — Token grants `canPublish + canSubscribe`, identity `role-userId`, TTL 1h

**Decision:**

```ts
// src/infrastructure/livekit/livekit-server.ts — internal helper
const at = new AccessToken(LIVEKIT_API_KEY, LIVEKIT_API_SECRET, {
  identity: `${role}-${userId}`, // e.g. "doctor-<uuid>" or "paciente-<uuid>"
  ttl: "1h",
});
at.addGrant({
  roomJoin: true,
  room: roomName, // "cita-<uuid>"
  canPublish: true,
  canSubscribe: true,
  canPublishData: false, // in-call chat is out of scope; do not enable it prematurely
});
return at.toJwt();
```

**Rationale:** Both parties need mic+cam and view, which is the standard 1:1 medical call. The 1h TTL is the right tradeoff — most consultations are 15-30 minutes, but allowing 1h absorbs overruns without a re-fetch dance that would drop the call. The identity format makes the role visible in LiveKit's participant list ("doctor-X" / "paciente-Y" rather than two opaque UUIDs), which is what the call page's participant panel will surface. `canPublishData` stays off because in-call chat is explicitly out of scope (D-OOS).

**Alternative considered:** TTL 2h — rejected, doubles the token-leak window for no realistic gain. TTL 30min — rejected, too tight for any consult that runs long. The user can re-join (and re-fetch a token) if the call really does go past 1h.

### D3 — `protectedProcedure` + role-aware use case

**Decision:** `bookings.getRoomToken` uses the existing `protectedProcedure` (any authenticated user). The use case `getRoomTokenUseCase` does the fine-grained authorization: it loads the `Cita`, checks that the caller is EITHER the cita's doctor OR the cita's patient, and throws `FORBIDDEN` otherwise. It also enforces the status gate (D4) and writes the audit log.

**Rationale:** This is the existing pattern in `bookings.ts` — `protectedProcedure` for the auth boundary, the use case for domain rules. A separate `videoCallsRouter` would duplicate the auth context plumbing and orphan the procedure from the booking domain (see AD-1).

**Alternative considered:** Custom procedure with role-aware logic in the procedure — rejected. The use case is the testable, framework-free home for this authorization; the procedure is just the wire adapter.

### D4 — Status gate: `CONFIRMADA` within ±15 min OR `EN_CURSO`

**Decision:** The token is issued only when EITHER:

1. `cita.estado === EN_CURSO`, OR
2. `cita.estado === CONFIRMADA` AND `|now - cita.fechaHora| <= 15 minutes`

Anything else throws `FORBIDDEN` with a clear Spanish message that varies by case:

- `PENDIENTE` → "La cita debe estar confirmada antes de unirse a la videollamada."
- `CONFIRMADA` outside the window → "La videollamada se habilita 15 minutos antes de la hora de la cita."
- `COMPLETADA` / `CANCELADA` / `NO_ASISTIO` → "Esta cita ya no permite unirse a una videollamada."

**Rationale:** Respects the existing status machine — the call is not a separate state, it's a capability that becomes available at `EN_CURSO` and the brief pre-window. The 15-minute early window lets the doctor open the call a bit early to verify their setup before the patient arrives. The explicit `EN_CURSO` is the "call is live" state, included so the call survives a status transition that happened before the patient fetched the token.

**Alternative considered:** Open the call at `CONFIRMADA` regardless of time — rejected, a confirmed appointment 2 days from now should not let either party wander into a room. Open at `EN_CURSO` only, no pre-window — rejected, the doctor cannot get ready.

### D5 — Dev API key/secret = `devkey` / `secret`

**Decision:** The `livekit/livekit-server --dev` mode uses the official defaults `devkey` (key) and `secret` (secret). These are documented in the LiveKit repository and the `livekit-server --dev` help text. `.env.example` carries placeholders (`devkey` / `replace_me_in_local`); `.env.local.example` carries the real dev values (`devkey` / `secret`).

**Rationale:** The official default is `devkey` / `secret`, NOT `devsecret`. The `livekit/livekit-server --dev` command requires this exact pair, and any mismatch makes the container fail to start with a confusing "invalid api key" error that wastes an hour of debug time. Citing the official source: https://github.com/livekit/livekit (`README.md`, "Trying LiveKit" section). The `.env.example` keeps a placeholder so that committing real-looking secrets is not encouraged.

**Alternative considered:** Generate random keys per dev machine — rejected, every fresh clone would need a `docker compose run livekit` step to print the keys, breaking the "clone → up → works" flow.

### D6 — Dev URL = `ws://localhost:7880`

**Decision:** `NEXT_PUBLIC_LIVEKIT_URL=ws://localhost:7880` in dev. NOT `wss://` (the LiveKit dev server has no TLS). `ws://` works here because `localhost` is exempt from the browser's secure-context rule for WebRTC: even though `getUserMedia()` normally requires a secure context, `localhost` is treated as a secure context in every modern browser.

**Rationale:** Production needs `wss://` with a real cert, but the dev story stays simple. The browser's exemption for `localhost` is the only reason this works — if a dev tunnels via `ngrok` or runs the browser on a different host, they will hit a secure-context error. The dev doc on starting LiveKit (item 11 in the in-scope list) explicitly calls this out.

**Alternative considered:** Generate a self-signed cert for dev — rejected, adds an `mkcert` step to the bootstrap and `wss://` to the browser config for no functional benefit on `localhost`.

### D7 — Modality toggle deferred

**Decision:** All `CONFIRMADA` and `EN_CURSO` citas are video-capable in MVP. There is no `presencial` / `online` toggle, and there is no `citas.modalidad` column. The join button appears whenever the status and time gate pass.

**Rationale:** The modality toggle requires a separate schema change (`citas.modalidad` column + booking-flow UI to select it + doctor profile UI to mark which modalities they support). The MVP proves the call mechanics on top of the existing flow. The toggle can be added later as a filter on `JoinCallButton` visibility and a field on `createAppointment`.

**Alternative considered:** Make the join button opt-in per cita via a `citas.videocallHabilitada` boolean — rejected. That's a poor-man's modality toggle that we'd have to rip out when the real one lands.

### D8 — Page slug `/citas/[id]/llamada`

**Decision:** The call page lives at `/citas/[id]/llamada`, mirroring the existing Spanish URL style (`/citas`, `/doctores`, `/perfil`, `/doctores/[id]/agendar`). The `JoinCallButton` on the detail page navigates with `router.push(\`/citas/${citaId}/llamada\`)`.

**Rationale:** SEO and URL aesthetics. The site's URL space is Spanish-first; `/call/[id]` or `/video/[id]` would break the convention. Nesting under `/citas/[id]` keeps the URL hierarchy meaningful (a call is an action on a cita, not a top-level resource).

**Alternative considered:** Top-level `/llamada/[id]` — rejected. Loses the relationship between call and cita in the URL.

### D9 — Two chained PRs (PR-1 data + API, PR-2 UI + tests)

**Decision:** PR-1 ships the data plane (column + migration + domain getter + LiveKit client + use case + procedure + Docker + env + audit union + 3 new specs + 4 delta specs). PR-2 ships the UI plane (call page + join button + 1 new UI spec + 1 delta spec + 2 component tests). PR-2 depends on PR-1. The chain is `stacked-to-main` (cached user preference from engram): PR-1 merges to `main`, PR-2 stacks on top and merges to `main` after PR-1 lands.

**Rationale:** Total estimated size is ~1,220 lines, which exceeds the 800-line review budget. The split is clean — PR-1 is the data + domain + API, PR-2 is the UI; each is under 800 lines and independently shippable. This mirrors the chained-PR pattern from `home-page-upgrade` and `doctor-profile-page` (both cited as precedents). PR-1 alone proves the end-to-end token issuance can be tested with curl before the UI is built, which de-risks the LiveKit integration.

**Alternative considered:** Single PR — rejected, the 800-line cap exists for a reason. PR-1 alone and skip the chain — rejected, the call page is useless without the procedure.

### D10 — Known limitation: `EN_CURSO` can stick if both leave without action

**Decision:** The status machine does not auto-transition `EN_CURSO` → `NO_ASISTIO` / `COMPLETADA` when both participants leave the call. If the doctor and patient both close their browsers without the doctor clicking "Completar" or "No asistio" on `/citas/[id]`, the cita stays in `EN_CURSO` indefinitely. The limitation is documented in three places:

1. The `video-calls-ui` spec, as a "Known Limitations" requirement.
2. The call page itself, as a small footer note: "Si ambos salen sin finalizar, la cita quedará en 'En curso' y el doctor deberá cerrarla manualmente desde /citas/[id]."
3. The change's archive report, so the next reviewer scanning the diff sees it.

**Rationale:** Closing the gap requires a LiveKit webhook receiver (`room_finished` event) that calls `updateAppointmentStatus` with `NO_ASISTIO` after a grace period. That is a real piece of work (webhook endpoint, signature verification, idempotency, retry handling) and is a separate change ("livekit-webhooks"). Documenting the limitation is the honest move — pretending it does not exist in MVP would make the call page look broken in production.

**Alternative considered:** Implement the webhook in this change — rejected, doubles the change size and crosses the budget by a wide margin.

## Stakeholders / Impact

| Stakeholder | Impact |
|---|---|
| **PACIENTE** (end user) | **Positive.** First-time patients who cannot or will not travel can now meet their doctor in the platform. Reduces no-shows for follow-ups (currently the only way to "meet" is to show up in person or use a third-party app). Patient identity in the call is `paciente-${userId}` — visible in the participant panel, no PII leak. |
| **DOCTOR** (verified profile owner) | **Positive.** Runs telehealth from the platform; no context-switch to Zoom or Google Meet. Can review notes, status, and the call in one window. The early-join window (D4) lets the doctor verify their setup before the patient arrives. |
| **DPO / Compliance** | **Positive.** 1:1 calls, server-side token issuance, audit-logged. No third-party cloud for A/V traffic — LiveKit runs in our Docker. The audit log entry is by user, action, citaId, and role; the token itself is NEVER logged. |
| **ADMIN** | **Neutral.** No new admin functionality. Admin can see `EN_CURSO` citas stuck in limbo (D10) via the existing `getMyAppointments` filter, but the auto-cleanup is out of scope. |
| **Billing / Product** | **Forward-looking.** Opens the door to a paid telehealth offering (subscription, per-call billing) — explicitly out of MVP. The call has no price field; doctors set prices per service in the existing `doctor_servicios` table. |
| **Infra / DevOps** | **Net new.** One more Docker service to operate (`livekit`), three more env vars. In dev: trivially `docker compose up -d`. In production: needs TURN + TLS — out of scope for this change. |
| **SEO / Marketing** | **None.** The call page is auth-gated. No public traffic. |
| **`/citas/[id]` (existing detail page)** | **Indirect.** One new component (`JoinCallButton`) added. No existing layout change. |
| **`/citas/[id]/agendar` (booking page)** | **None.** The modality toggle that would change this page is out of scope (D7). |
| **`DoctorHero` (existing doctor profile component)** | **None this change.** The `tel:` "Llamar" button stays as-is. Disabling it is a follow-up. |

| 3 | **Token expires mid-call** (1h TTL) — the patient is still in the call, but the JWT is past expiry. The WebRTC session may drop. | Low | High | The 1h TTL is the right tradeoff for the typical 15-30 minute consult. The SDK auto-refreshes the connection while the room is active — the JWT expiry matters only at room join, not while already inside. A future change can add a re-fetch pattern that the call page calls at the 50-minute mark. |
| 4 | **Patient joins before doctor is ready** — patient lands in an empty room. Awkward. | Medium | Low | The 15-minute pre-window (D4) limits this. The call page shows a "Esperando al doctor..." state (similar to the existing waiting patterns in the app) until a second participant joins. After 2 minutes alone, the page shows a polite "El doctor aún no se ha unido" message. |
| 5 | **Status machine broken: both leave without "Completar"** — D10 limitation. `EN_CURSO` cita stuck. Doctor's dashboard shows it as "in progress" forever. | Medium | Medium | Documented in three places (spec, call page footer, archive report). Mitigated in the future by a `livekit-webhooks` change. For MVP, doctors can manually transition from `/citas/[id]`. |
| 6 | **Network firewall blocks UDP** — many corporate networks block UDP. WebRTC falls back to TURN, which the dev container does not serve. | Low | Medium | Out of MVP. Documented as a production concern in the `livekit-infrastructure` spec's "Known Limitations" requirement. The dev experience is unaffected. |
| 7 | **Token leak in audit log** — the `CITA_ROOM_TOKEN_ISSUED` audit entry accidentally includes the JWT. Long-term credential leak. | Low | High | **D2 + D3 enforcement.** The audit `detalles` jsonb is built from a fixed shape `{ citaId, roomName, role }` — the token value is never threaded through the use case return value into the audit call. The `livekit-server.ts` wrapper returns `{ token, identity, roomName, expiresAt }` and the use case destructures only what it needs. The audit call site is a single line in the procedure, reviewed for the leak during PR-1. |
| 8 | **Two patients in the same room** — a malicious user tries to call `getRoomToken` for someone else's cita. | Low | High | The use case loads the cita, checks the caller is EITHER the cita's `doctorId` (via `doctor.userId === callerId`) OR the cita's `pacienteId` (via `paciente.userId === callerId`). Anyone else gets `FORBIDDEN`. The room name is `cita-${uuid}` (D1), so even a successful guess of a room name yields a valid token only for that one cita's two parties. |
| 9 | **`livekit-server-sdk` version drift** — the SDK has had JWT shape changes between minor versions. Pin to exact version in `package.json`. | Low | Low | Use exact version (no `^`): `"livekit-server-sdk": "2.x.y"`. The package.json update is in the PR-1 review scope. |
| 10 | **Browser secure-context rule for non-localhost** — a dev tunnels via `ngrok` and hits "secure context required" because `wss://` is not used and the hostname is not `localhost`. | Low | Low | D6 + dev doc. Out of MVP to make `mkcert` work. The dev doc tells devs to use `localhost` and warns against tunneling for video-call work. |

## Rollback Plan

1. **Revert the PR** — the data plane (PR-1) is fully behind one merge commit; the UI plane (PR-2) is a second. Reverting both restores the pre-change state.
2. **Drop the column** — the migration is additive (`ALTER TABLE ... ADD COLUMN ... DEFAULT NULL`). Rollback drops the column with `ALTER TABLE citas DROP COLUMN livekit_room_name`. No data loss because the column is unused (the getter derives the name from `cita.id`, so the column is dead weight by design — see D1).
3. **Remove the Docker service** — `livekit` is one service block in `docker-compose.yml`. Deleting the block + `docker compose down` removes it. No volume to clean.
4. **Remove the env vars** — three lines in `.env.example` and three in `.env.local.example`. The app boots without them (the LiveKit client wrapper throws at first use, not at boot).
5. **Delete the procedure** — `getRoomToken` has no consumers outside the call page (PR-2). Removing the procedure has no blast radius.
6. **Delete the call page + button** — `src/app/citas/[id]/llamada/` and `JoinCallButton.tsx`. The detail page falls back to the pre-change layout.
7. **Remove the `livekit-server-sdk` dep** — `pnpm remove livekit-server-sdk`. The client-side `@livekit/components-react` and `livekit-client` are left in place (they are tiny and may be reused; removing them is a separate cleanup PR).
8. **Verify `/citas/[id]`** renders without the button. No other page in the app is touched.

**No data loss in any rollback step.** The `livekit_room_name` column is reserved for future use and is never written to in MVP.

## Architecture Decisions

### Mini-ADR table

| ID | Decision | Rationale | Alternatives Considered |
|---|---|---|---|
| **AD-1** | New tRPC procedure `bookings.getRoomToken({ citaId })` in the existing `bookings` router — NOT a new `video-calls` router. | Video calls ARE bookings; the procedure is the natural extension of the booking flow. The auth context (which user is calling, what role, what session) is identical to the rest of `bookingsRouter`. A separate router would create a cross-router dependency for the auth context (we'd need to share the `protectedProcedure` middleware and the `findDoctorByUserId` / `findPacienteByUserId` helpers). | New `video-calls` router — rejected, orphans the procedure from the booking domain and duplicates the auth context. Reuse `bookings` router's `createAppointment` — rejected, mixing the data-mutation flow with the read-only token issuance would muddy the procedure list. |
| **AD-2** | LiveKit server client wrapper at `src/infrastructure/livekit/livekit-server.ts` — NOT in the use case. | Separation of concerns — the use case orchestrates (auth, status gate, time, audit), the LiveKit client is a thin wrapper around the SDK's `AccessToken` and `RoomServiceClient`. The use case is testable without mocking the SDK. The wrapper is a single class with a constructor that reads env vars, a `createAccessToken` method, and a `getRoomService` accessor for future use (recording, webhooks). | Put the `AccessToken` call in the use case — rejected, couples domain logic to a third-party SDK and makes the use case mock-heavy in tests. Put the wrapper in `src/infrastructure/api/` — rejected, `livekit-server.ts` is an SDK adapter, not an API adapter. |
| **AD-3** | Server-side room name derivation (`cita-${uuid}`), NOT stored in the DB column. The column is added for future flexibility but the getter always derives from `cita.id`. | Prevents client-side leaks of the naming convention (D1). Allows the column to be repurposed later (explicit room naming, persistence, audit trail) without breaking the API. The getter on `Cita` is a one-liner that never returns null (unlike a DB column that would need a fallback for the `NULL` case). | Store the room name explicitly on every cita — rejected, adds write overhead on `createAppointment` and creates a second source of truth that can drift. |
| **AD-4** | No new shadcn components. Reuse `Button`, `Card`, `Spinner` (or a Tailwind `animate-pulse`), and the `Video` icon from `lucide-react`. | The call page is mostly LiveKit components (`<LiveKitRoom>`, `<VideoConference>`, `<ControlBar>`); the wrapper UI is minimal. Adding new shadcn components for a 3-element wrapper would be review overhead with no win. The `JoinCallButton` is a single `<Button>` with conditional render. | A custom "call panel" component — rejected, `VideoConference` already provides the call panel. A custom loading skeleton — rejected, `Spinner` + `animate-pulse` covers it. |
| **AD-5** | Call page is a client component (`"use client"`). The page calls `api.bookings.getRoomToken.useQuery()` from the client; the page does NOT pre-fetch the token server-side. | Tokens are short-lived (1h) and the call is a client-side interaction. Pre-fetching server-side would require a complex redirect dance (server fetches token, stuffs it into a query param, client reads it from the URL — leaky and ugly). The client-side `useQuery` gets a fresh token on every page mount, which is the right semantic. | Server-side token fetch in the RSC — rejected, the token is sensitive and would be in the HTML payload. Server-side fetch + cookie handoff — rejected, over-engineered for a 1h token. |
| **AD-6** | Two chained PRs (PR-1 data + API, PR-2 UI). PR-1 ships the data plane (column, migration, domain getter, LiveKit client, use case, procedure, Docker, env, audit union, 3 new specs, 4 delta specs, 3 tests). PR-2 ships the UI plane (call page, join button, 1 new UI spec, 1 delta spec, 2 component tests). | Each PR is under 800 lines (PR-1 ≈ 580, PR-2 ≈ 640). The data plane can be reviewed and tested independently from the UI. PR-1 alone proves end-to-end token issuance with curl before the UI is built, de-risking the LiveKit integration. The chain is `stacked-to-main` (PR-1 merges to `main`, PR-2 stacks on top). This mirrors the pattern from `home-page-upgrade` and `doctor-profile-page`. | Single PR — rejected, the 800-line cap exists for review focus. Three PRs (data, API, UI) — rejected, the API is not useful without the data and the data is not useful without the API. The 2-way split is the natural boundary. |
| **AD-7** | Dev mode (`--dev --bind 0.0.0.0`) with no TLS. The browser connects via `ws://localhost:7880`. Production requires `wss://` + real certs (out of MVP). | Simplicity for dev. `localhost` is exempt from the browser's secure-context rule for WebRTC, so `ws://` works. The `--dev` flag enables permissive defaults (no auth on the management API, in-memory room store) that are fine for `localhost` and unsafe for production. The `livekit-infrastructure` spec makes the dev/prod boundary explicit. | Self-signed TLS for dev — rejected, adds `mkcert` to bootstrap for no functional benefit on `localhost`. Production-like config in dev — rejected, the dev experience should be "clone → up → works". |
| **AD-8** | Status machine: keep the existing flow. Doctor clicks "Iniciar consulta" → `EN_CURSO`. Both join the call. Doctor clicks "Completar" → `COMPLETADA`. The video call is a feature ON TOP of the status machine, not a replacement. | Respects the existing design (the status machine is the source of truth for billing, audit, and the doctor's dashboard). A new "CALL_ACTIVE" state would be redundant — `EN_CURSO` already means "the call is happening". The call is gated by the status (D4) but does not own the state. | Replace the status machine with a call-aware state — rejected, would require a status migration on every existing `EN_CURSO` cita and break the audit log. Add a separate `callActive: boolean` to `citas` — rejected, the same information is already in `estado`. |
| **AD-9** | `JoinCallButton` is a separate component at `src/components/citas/JoinCallButton.tsx`, not inline in `/citas/[id]`. | The visibility logic (status + time window + role-aware label) is non-trivial and benefits from a focused component with its own test suite. The detail page stays a server component; the button is the only client island on the page. | Inline the button in the page — rejected, the visibility test would be coupled to the page's other logic and harder to maintain. Make the page a client component — rejected, the page is a server-rendered layout and making it client-side would lose the SEO/auth benefits. |
| **AD-10** | Audit log entry for `getRoomToken` includes `citaId`, `roomName`, `role` — NOT the token. | Secrets do not go in audit logs. The audit log is persisted in `audit_logs.detalles` (jsonb) and is potentially readable by admins and DPOs. The token is a 1h credential; logging it would be a 1h credential leak per audit log query. The audit entry is built from a fixed shape in the use case's return; the use case never returns the token to the caller (the token is handed back to the client via the tRPC response, separately from the audit call). | Include a redacted version of the token — rejected, redacted tokens are still reconnaissance value for an attacker. Omit the audit log entirely — rejected, the compliance requirement is to log every token issuance. |
