# Proposal: Appointment Modality (Presencial / Online)

## Change Name

`modality-toggle` (folder: `2026-06-19-modality-toggle`)

## Intent

The platform already advertises online consultations as a feature — `terminos/page.tsx` line 36 promises citas "presenciales como en modalidad de videollamada" and `home-ui/spec.md` line 256 echoes the same — but the data model never grew the columns to back it up. `citas` has no `modalidad` column, `doctores` has no opt-in flag, `JoinCallButton` fires on every `CONFIRMADA/EN_CURSO` cita regardless of intent, and `getRoomToken` happily issues a LiveKit JWT to a cita that was never meant to be online. The video-calls change (`2026-06-16`, archived) shipped the call mechanics; this change ships the **business rule** that decides which citas get them.

The change is **additive at the DB layer and the spec layer** (a new enum column, a new boolean, a new tRPC procedure, new enum value), **non-destructive at the UI layer** (the booking flow gets one extra picker; the cita detail page keeps everything it has), and **deliberately minimal at the doctor surface** (a single toggle in `/configuracion` Preferencias — no new page, no per-day grid). It also closes the **`tel:` "Llamar" footgun on `DoctorHero`** that the video-calls change deliberately deferred, because a doctor who toggles on online consultations should not be advertising an off-platform phone number as the next best thing.

The previous change (`2026-06-16-video-calls`, archived) shipped the call page, the `getRoomToken` procedure, the audit union, and the LiveKit wiring. This change reuses all of it and adds the gating rules. The previous change's D7 ("modality toggle deferred") is now obsolete; this change consumes that promise.

## Why

The explore phase (engram id 453, 2026-06-19) found six concrete gaps. Every one is a real surface in the running app that contradicts what the marketing copy and the in-platform UI already say:

1. **DB schema gap** — `citas` (32 lines, `src/infrastructure/db/schema/citas.ts`) has no `modalidad`; `doctores` (29 lines, `src/infrastructure/db/schema/doctores.ts`) has no `acepta_online`. The `citas.livekit_room_name` column added by video-calls is still never written to at runtime (the room name is server-derived).
2. **Booking flow gap** — `src/app/doctores/[id]/agendar/page.tsx` (172 lines) and `SlotGrid` go straight from "pick slot" to "create cita with motivo" with no modality question. The patient is committing to a cita before the doctor has ever confirmed they offer online.
3. **`getRoomToken` gap** — `src/infrastructure/api/routers/bookings.ts` (~413 lines) and `src/application/use-cases/bookings/get-room-token.use-case.ts` (the 6th procedure in the bookings router) check only `estado` and the time window. A `CONFIRMADA` PRESENCIAL cita 5 minutes before `fechaHora` can already fetch a JWT. The token gate is the only thing standing between a presencial cita and an off-platform call.
4. **`JoinCallButton` gap** — `src/components/booking/JoinCallButton.tsx` (56 lines) and `src/app/citas/[id]/page.tsx` (408 lines, rendered in BOTH the DOCTOR and PACIENTE views) show the join CTA for any cita that passes the status/time check. PRESENCIAL citas in the next 15 minutes get a "Unirse a la videollamada" button that leads to a `FORBIDDEN` page.
5. **Doctor profile gap** — `src/components/profiles/DoctorHero.tsx` (171 lines) has a `tel:` "Llamar" button that fires whenever `telefonoConsulta` is set, and zero "Disponible online" affordance. A patient scanning three doctors in `/doctores` cannot tell which ones offer video calls.
6. **Doctor settings gap** — `src/app/configuracion/page.tsx` (129 lines) and `src/app/dashboard/disponibilidad/page.tsx` (405 lines) are the two existing surfaces. Neither has a toggle for online consultations. Doctors who want to opt in or out have no in-platform lever.

The change is small in absolute terms (~915 lines estimated) but high-leverage: it gates the entire video-call surface on a real business rule and unlocks the marketing promise the platform already makes.

## Default Decisions (D1–D13)

These 13 decisions are committed. The spec, design, apply, and verify phases MUST honor them verbatim.

### D1 — `citas.modalidad` is `varchar(20) NOT NULL DEFAULT 'PRESENCIAL'`

**Decision:** `modalidad: varchar('modalidad', { length: 20 }).notNull().default('PRESENCIAL')` in the Drizzle schema. The migration adds the column WITH a default, then drops the default in a second statement (see D3) so new inserts are explicit.

**Rationale:** `NOT NULL` is the right business invariant — a cita MUST have a modality. The default backfills every existing row in production to `PRESENCIAL` (the only safe historical assumption: there was no online modality before this change), and a Drizzle `default('PRESENCIAL')` keeps dev and test inserts ergonomic. `varchar(20)` matches the existing pattern (`estado` is `varchar(20)` too) and gives room for a third modality in the future ("DOMICILIO"?). The two-statement migration is the standard Postgres pattern for "backfill, then enforce".

**Alternative considered:** Nullable column + explicit backfill query — rejected. The default-backfill is atomic with the ADD COLUMN in Postgres ≥ 11 (it acquires an `ACCESS EXCLUSIVE` lock and rewrites the table in one pass), so there's no advantage to a nullable column + a separate `UPDATE` for the data we already know is `PRESENCIAL`. A `pg_enum` column was considered — rejected; the codebase uses `varchar` + TS string enums everywhere and adding a `pg_enum` for one field is a stylistic outlier.

### D2 — `doctores.acepta_online` is `boolean NOT NULL DEFAULT false` (opt-in)

**Decision:** `acepta_online: boolean('acepta_online').notNull().default(false)`. Mirrors the existing `verificado: boolean('verificado').default(false).notNull()` pattern on the same table.

**Rationale:** Opt-in is the safe default. Doctors who have never expressed an opinion on online consultations should not have video calls offered in their name. A flip to `true` is an explicit, audit-logged action. The flag is at the doctor level (per-practice), not per-day or per-slot (see OQ2 — the explore report deferred per-day modality to a follow-up).

**Alternative considered:** Opt-out (default `true`) — rejected. The default `false` matches the marketing posture of the platform (the live app does not yet have a single online cita) and avoids a privacy surprise for doctors who never opted in. Nullable — rejected; the only valid values are `true` and `false`, and a nullable boolean is a code smell.

### D3 — Migration `0004` in 2 SQL statements

**Decision:**

```sql
-- Statement 1: add column with default, backfills existing rows in one pass
ALTER TABLE "citas" ADD COLUMN "modalidad" varchar(20) DEFAULT 'PRESENCIAL' NOT NULL;

-- Statement 2: drop the default so new inserts must be explicit
ALTER TABLE "citas" ALTER COLUMN "modalidad" DROP DEFAULT;
```

**Rationale:** Postgres ≥ 11 backfills the new column in the same statement that adds it — no separate `UPDATE` step, no second pass, no risk of forgetting a row. After the backfill, dropping the default forces every new insert to specify a modality, which is the right invariant: the use case is the single place that picks a modality, and accidental `INSERT`s from a future migration or admin script will fail loudly instead of silently defaulting to `PRESENCIAL`. Same pattern is applied to `doctores.acepta_online` (backfill to `false`, then `DROP DEFAULT`).

**Alternative considered:** Single statement with `NOT NULL` and a permanent default — rejected. The permanent default is a footgun (see D1). Add column nullable, `UPDATE` to set, then `ALTER ... SET NOT NULL` — rejected, three statements where two suffice.

### D4 — Modality picker is AFTER the slot pick, BEFORE motivo consulta (OQ1=b)

**Decision:** The booking flow on `/doctores/[id]/agendar` becomes a three-step wizard rendered on a single page:

1. **Pick slot** — the existing `SlotGrid` (no change).
2. **Pick modality + write motivo** — a new section that appears after a slot is selected. Two-button toggle (or radio group): `Presencial` (always enabled) and `Online` (disabled with a tooltip "Este doctor no ofrece consultas online" when `doctor.aceptaOnline === false`). Below the modality toggle, the existing motivo textarea.
3. **Confirm** — single "Confirmar reserva" button that calls `bookings.createAppointment` with both `modalidad` and `motivoConsulta`.

**Rationale:** Slot pick first because the slot is the most-constrained thing the patient is choosing (the doctor has limited availability, and the patient usually has only a few windows). Modality is the second-most-constrained (only available if the doctor opted in). Motivo is the least-constrained (any non-empty string), so it goes last. Combining modality + motivo into a single confirm step (the user explicitly chose this over a four-step wizard with a separate "Review" step) keeps the flow short — three interactions, three clicks — and gives the doctor everything they need to triage the cita at a glance.

**Alternative considered:** Modality BEFORE slot pick — rejected (OQ1=a). A doctor who doesn't offer online would force the patient to pick a modality only to learn that the only available slot is incompatible. Modality pick in a separate "Review" step after motivo — rejected (OQ1=c). Adds a click and a screen for no information gain; the modality picker is two buttons, not enough to justify a page.

### D5 — Booking validation rejects `ONLINE` when `doctor.aceptaOnline === false`

**Decision:** Inside `createAppointmentUseCase`, after the existing `doctorId/fechaHora/motivoConsulta/availability/conflicting-cita` checks, add:

```ts
if (input.modalidad === "ONLINE") {
  const doctor = await tx.query.doctores.findFirst({
    where: eq(schema.doctores.id, doctorId),
    columns: { aceptaOnline: true },
  });
  if (!doctor?.aceptaOnline) {
    throw new TRPCError({
      code: "BAD_REQUEST",
      message: "El doctor no ofrece consultas online",
    });
  }
}
```

**Rationale:** Server-side enforcement is the only enforcement that matters — a determined patient can mutate the React state to enable the Online button. The check is inside the same transaction as the cita insert, so a doctor who toggles `aceptaOnline` to `false` between the patient's "Confirmar" click and the mutation is correctly caught (worst case the patient sees "El doctor no ofrece consultas online" and re-picks `PRESENCIAL`).

**Alternative considered:** Client-side disable only — rejected. The same UX (disabled button) but no server enforcement, which means a curl request bypasses the rule. The disabled button is still rendered (D16) for good UX, but it's a hint, not the guard.

### D6 — `getRoomToken` gate rejects `PRESENCIAL` with a clear Spanish message

**Decision:** Inside `getRoomTokenUseCase`, after the existing `estado + time window` check, add:

```ts
if (cita.modalidad === "PRESENCIAL") {
  throw new TRPCError({
    code: "FORBIDDEN",
    message: "Esta cita es presencial, no permite videollamada",
  });
}
```

**Rationale:** The existing use case already loads `cita` (with `doctores` and `pacientes` joined in a single query); the `modalidad` column is now part of that load. The gate is a one-line `if`. The error message is distinct from the existing status/time-window messages so the caller (the call page) can render the right copy: "Esta cita es presencial" is a different fix path than "La cita debe estar confirmada". The token is never issued for a PRESENCIAL cita under any circumstance.

**Alternative considered:** No gate (let the call page handle the empty state) — rejected. The `getRoomToken` procedure is the security boundary; a PRESENCIAL cita must never receive a token. The call page rendering an empty state is a defense-in-depth, not a replacement. Different error code (`BAD_REQUEST` vs `FORBIDDEN`) — rejected; `FORBIDDEN` is the right semantic ("you don't have permission to do this"), `BAD_REQUEST` implies a malformed request.

### D7 — `JoinCallButton` requires a `modalidad` prop and returns `null` for PRESENCIAL

**Decision:** Add a required `modalidad: ConsultaModalidad` prop to `JoinCallButton`. The visibility rule becomes:

```ts
if (modalidad === "PRESENCIAL") return null; // hard gate, no fallback
const isOpen = estado === "EN_CURSO" ||
  (estado === "CONFIRMADA" && Math.abs(now.getTime() - fechaHora.getTime()) <= 15 * 60 * 1000);
return isOpen ? <Button>Unirse a la videollamada</Button> : null;
```

**Rationale:** The button is the user-facing manifestation of the `getRoomToken` gate. If the cita is PRESENCIAL, the button should never appear — not be disabled, not show a tooltip, not link to a 403 page. Returning `null` is the cleanest expression of "this UI doesn't apply". The call page already handles the unauthenticated / cancelled / past-cita cases the same way.

**Alternative considered:** Render the button disabled with a tooltip "Esta cita es presencial" — rejected. A disabled button that goes nowhere is worse than no button; the modality badge on the cita detail page (item 18) already tells the user this is a presencial cita. Hide but log a warning when the procedure is called for PRESENCIAL — rejected; the use case throws `FORBIDDEN` and the procedure caller (the call page) already handles `FORBIDDEN` from the existing status gates.

### D8 — Doctor toggle lives in a new "Modalidad de consulta" card in `/configuracion` Preferencias (D12)

**Decision:** A new `<Card>` titled "Modalidad de consulta" appended to the existing `/configuracion` page, inside the "Preferencias" section. Contains:

- A single shadcn `<Switch>` (or a hand-rolled toggle if `<Switch>` is not yet in the project) labelled "Acepto consultas online".
- A help text below: "Aceptar consultas online habilita la opción de videollamada en el perfil público y en la agenda de los pacientes."
- An optimistic update via `api.profiles.updateAcceptsOnline.useMutation()`; on error, revert and show a sonner toast.

The toggle is at the doctor level (per-practice), NOT per-day (OQ2 deferred to a follow-up — see Out of Scope).

**Rationale:** `/configuracion` is the natural home for account-level preferences ("Tema" already lives there). `/dashboard/disponibilidad` is the schedule surface and conflating a one-time preference with a weekly grid is a UX smell. A new dedicated page is over-engineering for a single toggle. The help text is explicit about the consequence (public profile + booking flow) so doctors don't toggle on and then wonder why their phone keeps ringing.

**Alternative considered:** Fold the toggle into `updateProfile` (D12) and render it inside the `/perfil` edit form — rejected. The `/perfil` surface is for identity / professional info; a business-rules toggle is a different concern. The audit log benefit (D9) is also stronger when the toggle is its own procedure.

### D9 — Audit log: `DOCTOR_ACEPTA_ONLINE_CHANGED` (no `CITA_MODALIDAD_CHANGED` in MVP)

**Decision:** Extend the `AuditAction` union with `'DOCTOR_ACEPTA_ONLINE_CHANGED'`. The `detalles` jsonb is `{ aceptaOnline: boolean }`. NO `CITA_MODALIDAD_CHANGED` action is added in MVP — there is no post-creation modality change flow (OQ4 deferred to a follow-up).

**Rationale:** The audit log is the platform's compliance and debugging surface for state changes that affect what patients see. The doctor toggle is a state change with user-visible consequences (the doctor's public profile, the booking flow for new citas, the visibility of "Disponible online"). A cita's modality is set once at creation and never changes, so it doesn't need its own audit entry — the existing `CITA_CREADA` (or whatever the create event is called) entry covers it. If post-creation modality change ships in a follow-up, `CITA_MODALIDAD_CHANGED` lands with that change, not here.

**Alternative considered:** Add a generic `MODALIDAD_CHANGED` action that covers both — rejected. Different actors, different scopes (doctor vs patient), different consequences; one action per intent is easier to grep in the audit log.

### D10 — Chained PRs (stacked-to-main): PR-A data + settings + badge + listing filter, PR-B booking flow + JoinCallButton + getRoomToken

**Decision:** PR-A and PR-B are stacked. PR-A merges to `main`. PR-B branches from `main` after PR-A lands, stacks on top, and merges to `main`. (See "PR split recommendation" below for the per-PR diff scope.)

**Rationale:** Total estimated size is ~915 lines, which exceeds the user's 800-line review budget. The split is natural: PR-A is the "opt-in" surface (data + doctor toggle + DoctorHero badge + listing filter), PR-B is the "gating" surface (booking flow + JoinCallButton + getRoomToken). PR-A is independently shippable — the badge + toggle work even before the booking flow picker lands (existing bookings default to `PRESENCIAL`). PR-B is not shippable without PR-A (the `modalidad` column must exist). The chain is `stacked-to-main` per cached engram preference and the `chained-pr` skill.

**Alternative considered:** Single PR — rejected (915 > 800). PR-A only and skip the chain — rejected; the booking flow picker is the patient-facing reason this change exists, and shipping the doctor opt-in without the patient-side picker is a half-built feature.

### D11 — `listDoctorProfiles` accepts optional `aceptaOnline?: boolean`; `/doctores` shows a filter pill (OQ5=b)

**Decision:** `listDoctorProfiles` input is extended with `aceptaOnline?: boolean`. When set, the SQL adds `WHERE doctores.acepta_online = true`. The `/doctores` page renders a "Disponible online" pill (a clickable `<Badge variant="outline">` that toggles `?aceptaOnline=true` in the URL via `useRouter().replace()` and `useSearchParams`); when the URL has the param, the pill is filled and the query is sent with `aceptaOnline: true`.

**Rationale:** Patients browsing `/doctores` want to find online-capable doctors without opening every profile. The pill is a one-click filter that mirrors the existing `especialidad` search input style. URL-driven state means the filter is shareable and survives a page reload — a small but real UX win.

**Alternative considered:** Dropdown filter next to the search input — rejected. The single most common filter ("online?") deserves its own affordance, not a buried option in a dropdown. Checkbox in the search form — rejected; the pill is more discoverable and matches the "badge as a button" pattern already used in the project (the rating badge on `DoctorCard`).

### D12 — New `updateAcceptsOnline` tRPC procedure, NOT folded into `updateProfile`

**Decision:** Add `profiles.updateAcceptsOnline: protectedProcedure.input(z.object({ aceptaOnline: z.boolean() })).mutation(...)` as a new procedure. DOCTOR-only. The existing `updateProfile` is untouched.

**Rationale:** Three reasons not to fold into `updateProfile`:

1. **Cleaner audit log** — `updateAcceptsOnline` writes a single, well-typed `DOCTOR_ACEPTA_ONLINE_CHANGED` entry with `{ aceptaOnline: true }` as `detalles`. Folding it in would force `updateProfile` to compare old vs new values for `aceptaOnline` and emit a `PROFILE_FIELD_CHANGED` entry — more code, less specific.
2. **Simpler schema** — `updateProfile` is the all-purpose "update my doctor row" mutation. Adding a one-field procedure is the inverse: the schema is one field, the type is exact, the error surface is small.
3. **No pollution of the existing update** — `updateProfile` is exercised by every profile edit. Adding a new field to its input schema bumps its type signature and forces every existing caller to be reviewed for regression. A separate procedure isolates the change.

**Alternative considered:** Fold into `updateProfile` — rejected, see above. Make the toggle a server action (Next.js) — rejected. Server actions are fine for the form, but the audit log entry is the same either way, and a tRPC procedure is consistent with the rest of the platform.

### D13 — `getRoomToken` modality gate test covers BOTH the new rejection AND the existing happy path

**Decision:** The `get-room-token.test.ts` test extension adds TWO new scenarios:

1. `PRESENCIAL cita → FORBIDDEN "Esta cita es presencial, no permite videollamada"`
2. (Existing) `ONLINE cita within time window → returns token` — MUST still pass after the new gate is added

And the integration test (`bookings.getRoomToken.test.ts`) adds the same `PRESENCIAL → FORBIDDEN` scenario at the procedure level.

**Rationale:** The modality gate is added AFTER the existing status/time gate, so the new test must prove the gate works AND the existing happy path didn't regress. A common bug class is "added a new check, broke the existing flow because the existing flow was relying on a side effect the new check overwrites". The test for the existing happy path is the regression guard.

**Alternative considered:** Test only the new rejection — rejected. The existing happy-path test is already there (per the video-calls change), but it must be re-run after the modality column is added; if it relied on `cita.modalidad` being `undefined`, the test would now throw. Re-running it is non-negotiable.

## Architecture Decisions

| ID | Decision | Rationale | Alternatives Considered |
|---|---|---|---|
| **AD-1** | The modality picker is in the booking flow (D4), NOT on the doctor profile page. | Modality is a per-cita choice constrained by what the doctor allows. Asking it on the profile page (before the patient has picked a slot) is premature — the patient may not have a free slot at all. Putting it in the flow, after slot pick, makes the choice contextual. | Modality picker on `/doctores/[id]` (profile page) — rejected, asks the question too early. Modality picker on `/citas/[id]` after cita creation — rejected, the cita is already created and would need a modality change flow (OQ4 deferred). |
| **AD-2** | `citas.modalidad` is a `varchar(20)` with TS string union, NOT a Postgres `pg_enum`. | The codebase uses `varchar` + TS string unions for `estado` (`ConsultationStatus`), `rol` (`UserRole`), etc. Introducing a `pg_enum` for one field is a stylistic outlier and complicates the migration (adding a value requires `ALTER TYPE ADD VALUE`, which is non-transactional in older Postgres). | `pg_enum` — rejected, outlier. Nullable + TS check — rejected, `NOT NULL` is the right invariant. |
| **AD-3** | Two-statement migration (ADD COLUMN ... DEFAULT ... NOT NULL, then DROP DEFAULT) for `citas.modalidad` and `doctores.acepta_online`. | Postgres ≥ 11 backfills the new column in the same statement that adds it (atomic `ACCESS EXCLUSIVE` lock). After the backfill, dropping the default forces new inserts to specify a modality, catching accidental inserts in future migrations. | Single statement with a permanent default — rejected, leaves a footgun (any future `INSERT` silently defaults to `PRESENCIAL`). Three statements (nullable + UPDATE + SET NOT NULL) — rejected, two statements suffice. |
| **AD-4** | Modality + motivo are sent in a SINGLE `createAppointment` mutation (D4). | The user explicitly chose this over a four-step wizard. A single mutation is one round-trip, one transaction, one error path. The doctor sees the modality on the cita detail page immediately. | Two-step (create cita with `modalidad`, then `updateCita` with motivo) — rejected, two round-trips, two error paths, and a half-created cita in flight if the second call fails. |
| **AD-5** | The doctor opt-in toggle is a new tRPC procedure `profiles.updateAcceptsOnline` (D12), NOT a field in `updateProfile`. | Cleaner audit log (one specific `DOCTOR_ACEPTA_ONLINE_CHANGED` entry), simpler schema (one boolean), and no pollution of `updateProfile`'s wire surface (which is exercised by every profile edit). | Fold into `updateProfile` — rejected, see D12. Server action — rejected, tRPC is the project convention. |
| **AD-6** | `getRoomToken` rejects PRESENCIAL with a new `FORBIDDEN` message (D6), NOT with `BAD_REQUEST` or `NOT_FOUND`. | `FORBIDDEN` is the right semantic ("you don't have permission to do this") — the caller is authenticated, the cita exists, the only thing that's wrong is the modality. `BAD_REQUEST` implies a malformed request (the request is fine; the resource is the issue). `NOT_FOUND` would be misleading (the cita exists). | `BAD_REQUEST` — rejected, wrong semantic. `NOT_FOUND` — rejected, leaks no useful info but breaks the convention. Reuse an existing message like "Esta cita ya no permite unirse a una videollamada" — rejected, the modality is the WHY, not the state. |
| **AD-7** | `JoinCallButton` returns `null` for PRESENCIAL (D7), NOT a disabled button or a tooltip. | A disabled button that goes nowhere is worse than no button — the patient clicks, nothing happens, frustration. The modality badge on the cita detail page (item 18) already communicates "this is a presencial cita". | Disabled button with tooltip "Esta cita es presencial" — rejected, click-and-nothing is bad UX. Hidden but logged a warning — rejected, the use case throws and the call page already handles `FORBIDDEN`. |
| **AD-8** | The `tel:` "Llamar" button on `DoctorHero` stays, but a "Disponible online" badge is added (D11 / item 19). | A doctor who has explicitly set `telefonoConsulta` and ALSO opted into online is a real use case (e.g., they offer both, and the patient can choose). Removing the phone number is a doctor-level UX decision that requires follow-up. The badge gives patients a clear cue that online is available; the phone number is still a valid path for doctors who want to offer it. | Remove the `tel:` button when `aceptaOnline === true` — rejected, breaks a doctor-level UX choice and is not what the doctor asked for. Always show the badge — rejected, a doctor who has NOT opted in should not advertise online. |
| **AD-9** | `/doctores` filter pill is URL-driven (`?aceptaOnline=true`) (D11), NOT in-component state. | URL state survives a page reload, is shareable (a patient can send a link to a friend), and is the same pattern used by every other filter in the app. A `useSearchParams` + `useRouter().replace()` flow is the standard Next.js 15 idiom. | `useState` in the component — rejected, filter is lost on reload. localStorage — rejected, doesn't survive cross-device. |
| **AD-10** | Chained PRs (stacked-to-main), not a feature-branch chain (D10). | PR-A (data + settings + badge + listing filter) is independently shippable — the badge + toggle work even before the booking flow picker lands. PR-B (booking flow + JoinCallButton + getRoomToken) stacks on PR-A. A feature-branch chain with a tracker PR is overkill for two PRs and a clean linear dependency. | Single PR — rejected (915 > 800 budget). Feature-branch chain — rejected, the linear `PR-A → PR-B` dependency fits `stacked-to-main` better. |
| **AD-11** | The new "doctor-settings-ui" spec is scoped to MVP only (one toggle, one help string, optimistic update, audit). | The natural home for a doctor-side preferences spec. A narrow scope avoids over-promising (no notification toggles, no calendar sync, no language preferences) while leaving room for follow-up changes to add scenarios incrementally. | Don't add a new spec — rejected, the toggle surface is real and a spec makes it discoverable. Add multiple scenarios for hypothetical future settings — rejected, the spec is the contract for what ships now, not a wishlist. |
| **AD-12** | The booking page's modality picker is a two-button toggle (or radio group), NOT a `<Select>`. | Two options is the threshold for a toggle; a `<Select>` is for 3+. A toggle makes the choice obvious without an extra click to open the dropdown. | `<Select>` — rejected, 2 options is below the threshold. Three-state radio (presencial / online / either) — rejected, "either" is not a valid modality; it's what the patient picks. |
| **AD-13** | The `createAppointment` use case is wrapped in a single Drizzle transaction (existing pattern). The new `aceptaOnline` check runs INSIDE the transaction. | The transaction already does `SELECT FOR UPDATE` on conflicting citas and verifies the slot is within the doctor's availability. Adding the `aceptaOnline` check inside the transaction closes the TOCTOU window: a doctor who toggles `aceptaOnline` to `false` between the patient's "Confirmar" click and the mutation is correctly caught. | Outside the transaction — rejected, opens a TOCTOU window. Cache the doctor's `aceptaOnline` for the duration of the booking flow — rejected, requires an invalidation strategy and adds complexity. |
